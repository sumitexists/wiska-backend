This is the backend repository for the **Wiska Messaging Protocol** — a real-time, stateful messaging platform engineered from scratch rather than on top of a Backend-as-a-Service. It owns the network layer: persistent WebSocket connections, custom token-based auth over those connections, and the Postgres + Redis data/broadcast layer behind it.

For the React client, see the **[Wiska Frontend](https://github.com/sumitexists/wiska-frontend)** repo (live at [wiska.vercel.app](https://wiska.vercel.app)).

## Concept: One Account, Two Decoupled States

1. **Verified Profile** — a standard, authenticated REST identity for known contacts.
2. **The Anonymous Mask** — an untraceable, global broadcast identity. Users are assigned a hashed node ID that is completely decoupled from their public profile.

## 🛠️ Tech stack

| Layer | Choice |
|---|---|
| Web framework | Django 6.0 + Django REST Framework 3.17 |
| Auth | `djangorestframework-simplejwt` (JWT) + Djoser (auth endpoints) |
| ASGI server | Daphne |
| Real-time layer | Django Channels + `channels-redis` (Redis Pub/Sub as the channel layer) |
| Database | PostgreSQL, via `psycopg2-binary` + `dj-database-url` (Neon-compatible) |
| Cache | `django-redis` |
| API schema / docs | `drf-spectacular` (OpenAPI/Swagger generation) |
| Filtering & CORS | `django-filter`, `django-cors-headers` |

Wiska runs on a decoupled, asynchronous architecture so persistent connections never block the main thread.

## 🔐 Core engineering challenges solved

- **Custom JWT ASGI middleware** — standard HTTP auth headers don't exist during a WebSocket handshake, so a custom Django Channels middleware intercepts the raw connection payload, extracts the JWT from the query string, authenticates it against Postgres, and only then admits the connection to its Redis broadcast group.
- **Dual-identity schema decoupling** — the `User` table is kept structurally apart from the `AnonymousAlias` table using a Django-signal-driven generator, so a user's public face and private/anonymous identity can never be joined in a single API response.
- **N+1-free friend search** — the "add friend" search endpoints were rewritten with raw ORM exclusions and `values_list` flattening, cutting what would be N+1 queries down to one.

## 🚀 Local development setup

### Prerequisites

- **Python 3.12+** — Django 6.0 dropped support for Python 3.10/3.11
- A local or cloud instance of **PostgreSQL**
- A local or cloud instance of **Redis**

### Installation

**1. Clone and enter the directory**
```bash
git clone https://github.com/sumitexists/wiska-backend.git
cd wiska-backend
```

**2. Create and activate a virtual environment**
```bash
# Use uv or standard pip
python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate
```

**3. Install dependencies**
```bash
pip install -r requirements.txt
```

**4. Environment variables** — create a `.env` file in the root directory:
```env
SECRET_KEY=your_django_secret_key
DEBUG=True
DATABASE_URL=postgres://user:password@localhost:5432/wiskadb
REDIS_URL=redis://localhost:6379
FRONTEND_URL=http://localhost:5173
ALLOWED_HOSTS=localhost,127.0.0.1
```

**5. Run migrations and start the Daphne server**
```bash
python manage.py migrate
daphne -b 0.0.0.0 -p 8000 wiska_core.asgi:application
```

The backend is now listening for HTTP on port `8000` and WebSockets on `ws://127.0.0.1:8000`.

## 📡 API & WebSocket access

- All HTTP REST endpoints are protected by SimpleJWT bearer tokens.
- An OpenAPI/Swagger schema is generated by `drf-spectacular` — once the server is running, check `wiska_core/urls.py` for the exact schema/docs route in your version of the project.
- WebSocket routing lives alongside the real-time apps (`anonymous/`, `private/`) in their respective `routing.py` files.
